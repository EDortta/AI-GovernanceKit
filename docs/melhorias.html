O kit saiu de docs/ → .docs/

Uma reorganização de pastas que devolve a docs/ para o projeto e deixa a propriedade dos arquivos inequívoca. Aqui está o como, o porquê e o impacto.

layout do repositório
Antes

├─ software-overview.md
├─ limits.md
├─ workflows/ agents/

    └─ README.md

01

O que mudou

Uma frase: o kit deixou de ocupar a docs/ e passou a viver na .docs/. A docs/ agora é território exclusivo do projeto que adota o kit.

Kit

.docs/

Oculto e gerenciado. Sobrescrito no --upgrade. Nunca contém conteúdo escrito pelo projeto.

Projeto

docs/

Livre. Nunca é tocado pelo instalador. É onde ficam required-reading.md e a documentação do time.

Git

escolha

Versionar a .docs/ virou uma pergunta na instalação, salva em .governancekit.

02

Por que fizemos

Três dores reais do modelo anterior, em que o kit era dono da docs/.

Invasão

Projetos legados

Quem já usava docs/ como pasta própria via o --upgrade sobrescrever seus arquivos.

Confusão

Pasta dentro de pasta

A pasta do projeto (docs/project/) morava dentro da pasta do kit — invertendo a intuição de quem é dono.

Rigidez

Publicação no Git

Versionar (ou não) os docs do kit era decisão fixa. Uns times querem, outros não.

03

Como foi feito

Tudo no instalador Python (governancekit/install_agents.py). O ponto central é um mapeamento origem→destino, para o instalador funcionar mesmo antes de o repositório-fonte (AI-Agents) ser reestruturado. 

# Mapeia caminhos da origem (docs/…) para o destino (.docs/…),
# mantendo os seeds do projeto em docs/.
def _dest_rel(src_rel):
    if src_rel in _PROJECT_SEED_PATHS:        # required-reading, napkin
        return src_rel                        # fica em docs/
    if src_rel.startswith("docs/"):
        return ".docs/" + src_rel[len("docs/"):]   # vai para .docs/
    return src_rel
Fresh install

Sem herança

_FRESH_PATHS deixou de copiar docs inteiro. Copia caminhos explícitos, então um projeto novo não herda as issues do repositório-fonte.

Upgrade

Migração automática

_migrate_legacy_layout() move o kit de docs/ para .docs/, promove docs/project/* e cria backup. Idempotente.

Git

Escolha persistida

_resolve_track_kit_docs(): flag → config → pergunta → padrão. Uma única entrada .docs/ no .gitignore.

05

Migração para quem já adotou

Um comando. O instalador detecta o layout antigo e reorganiza com segurança.

governancekit install-agents --upgrade
  1. Backup primeiro. A docs/ inteira é copiada para .docs-migration-bak/ antes de qualquer movimento.
  2. Kit vai para .docs/. software-overview.md, limits.md, workflows/, agents/, issues/templates/ são movidos.
  3. Seus docs sobem. Tudo em docs/project/ é promovido para docs/; a pasta vazia é removida.
  4. Colisões são preservadas. Se um nome já existe na raiz, o arquivo legado fica no backup e é reportado — nada é sobrescrito.
  5. Rodar de novo não faz nada. Se a .docs/ já existe, a migração é ignorada.
